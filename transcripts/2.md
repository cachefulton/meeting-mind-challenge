## How Database Migration Works in This Starter

There are **two approaches** available to you, and the starter is actually set up for both:

### Approach A: `synchronize: true` (Zero-migration dev mode)

This is what's **already enabled**. Look at `data-source.ts` line 9:

9:9:starter/apps/api/src/database/data-source.ts
  synchronize: process.env.NODE_ENV !== 'production',


Since you're running in development (`NODE_ENV` is not `production`), TypeORM will **automatically create/alter database tables** to match whatever entity classes it finds. That means:

1. You create a `Meeting` entity file (e.g. `meeting.entity.ts`)
2. You start the dev server (`npx turbo dev`)
3. TypeORM sees the entity, diffs it against the actual database schema, and runs the necessary `CREATE TABLE` / `ALTER TABLE` statements automatically

**No migration files needed.** This is the fastest path for a time-boxed take-home.

### Approach B: Explicit migrations (via the TypeORM CLI)

The starter also has a migration pipeline wired up if you want it:

- **`data-source.ts`** exports a `DataSource` instance (line 14) which the TypeORM CLI uses
- **Migrations directory**: `starter/apps/api/src/database/migrations/` (line 8 of data-source.ts) — currently empty
- **Run script**: `db:migrate` in `starter/apps/api/package.json` runs `typeorm migration:run -d src/database/data-source.ts`
- **Root shortcut**: `npm run db:migrate` at the monorepo root delegates to the api workspace via Turborepo

The workflow for explicit migrations would be:

bash
# 1. Generate a migration from entity changes (run from starter/apps/api/)
npx typeorm migration:generate -d src/database/data-source.ts src/database/migrations/CreateMeeting

# 2. Run it
npm run db:migrate


### My Recommendation

**Use `synchronize: true`** (Approach A). The plan itself says this:

> With `synchronize` on in dev, you may skip migrations initially

For a 5-hour take-home, letting TypeORM auto-sync is the pragmatic choice. You just need to:

1. **Create the `Meeting` entity** file with your columns defined via TypeORM decorators
2. **Register it** in a `MeetingsModule` using `TypeORM.forFeature([Meeting])`
3. **Import `MeetingsModule`** in `AppModule`
4. **Add shared types** in `packages/shared` so the web app can reference the same shape

When you start the dev server, TypeORM will see the entity, create the `meetings` table in PostgreSQL, and you're done — no migration commands to run.

The `autoLoadEntities: true` in `app.module.ts` (line 10) means any entity registered via `forFeature()` in any module is automatically picked up, so you don't even need to manually list entities in the data source config for the running app.
